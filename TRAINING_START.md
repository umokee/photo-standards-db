# Training Start

Для текущего проекта лучше начинать не с полноценного UI и не с большого `train_model()`, а с маленьких шагов, которые можно проверить по отдельности.

## Общая идея

Постоянно хранить стоит только:

- исходные фото
- аннотации в БД
- базовые веса в `storage/models/_basic`
- готовые обученные веса в `storage/models/...`
- метаданные модели в БД

Постоянную папку с датасетами хранить не обязательно. Лучше перед обучением собирать временный dataset export, запускать тренировку и потом его удалять.

## С чего начать

Начинать лучше не с `train_model()`, а с маленьких методов.

### 1. Сбор данных из БД

Сначала сделать метод вроде:

- `get_training_source(db, group_id)`

Он должен собрать всё, что нужно для обучения одной группы:

- группу
- стандарты группы
- изображения стандартов
- сегменты стандартов
- аннотации изображений

Это первый шаг, потому что без него непонятно, хватает ли данных вообще.

### 2. Валидация данных для обучения

Потом метод вроде:

- `validate_training_source(source)`

Он должен проверить:

- есть ли изображения
- есть ли размеченные изображения
- есть ли сегменты
- есть ли хотя бы один класс
- нет ли пустых или битых аннотаций

Это нужно, чтобы обучение не падало позже странной ошибкой.

### 3. Сборка временного dataset export

Потом самый важный метод:

- `build_temp_dataset(source)`

Он должен:

- создать временную папку
- разложить `images/train`, `images/val`, `labels/train`, `labels/val`
- сделать `symlink` или `hardlink` на оригинальные фото вместо копирования
- сгенерировать `.txt` разметку
- создать `data.yaml`

С этого момента уже будет настоящий минимальный pipeline.

### 4. Конвертация аннотаций в формат YOLO

Эту часть лучше вынести отдельно, потому что она самая специфичная:

- `annotation_to_yolo_seg(points, image_width, image_height)`

или

- `build_yolo_label(image, annotations, segments)`

Этот код должен превращать полигоны из БД в строки формата YOLO segmentation.

### 5. Запуск обучения

Только после этого делать метод вроде:

- `run_training(dataset_dir, weights_path, params)`

Он должен:

- брать локальные базовые веса
- запускать обучение
- возвращать путь до лучшей модели, метрики и прочий результат

### 6. Сохранение информации о модели

После обучения:

- `create_model_record(db, group_id, training_result, params)`

Он должен сохранять:

- `weights_path`
- `architecture`
- `version`
- `epochs`
- `imgsz`
- `batch_size`
- `class_names`
- `metrics`

### 7. Оркестратор

И только в конце собирать:

- `train_model(db, data)`

Он уже просто вызывает всё по порядку:

1. `get_training_source`
2. `validate_training_source`
3. `build_temp_dataset`
4. `run_training`
5. `create_model_record`

## Минимальный набор методов для старта

Если совсем коротко, начни вот с этих методов:

- `get_training_source(...)`
- `validate_training_source(...)`
- `build_temp_dataset(...)`
- `train_model(...)`

А уже внутри `build_temp_dataset` потом добавишь helper-функции.

## Почему такой порядок

Так у тебя появляются понятные контрольные точки:

1. сначала видно, что данные вообще грузятся
2. потом видно, что валидация работает
3. потом получается готовый временный dataset
4. и только потом запускается тяжёлое обучение

## Практический принцип

Не делать `storage/datasets` архивом всех обучений.

Лучше считать источником истины:

- БД
- исходные фото
- аннотации

А dataset export делать временным рабочим представлением для конкретного запуска обучения.
